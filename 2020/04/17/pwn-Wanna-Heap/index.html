pwn: Wanna Heap
2020-04-17|Writeup
Word count: 3.9k|Reading time: 19 min

题目是HITCON在WCTF 2016给出的，好像是说当时没有队伍做出来（当然事实上现在给我做我也没办法自己做出来）。巧合的是，做这题之前正好在微信公众号上看到一个Ubuntu 17.04(Glibc 2.24)中libc任意地址写一个”\x00”字节的利用，给出的例题是WHCTF 2017的stackoverflow（能搜到官方wp）。最后花了两天时间，在这几篇文章提供的思路的基础上完成解题（其实完全复现的话应该不需要这么久，主要是我想看看能不能在既有的思路上有所变化，结果以失败告终）。

题目描述

x64位程序，保护全开：

整个题目的逻辑是：

  1. 首先用mmap，通过随机数随机分配一块内存，主要用于储存后续malloc出来的chunk指针，虽然分配的内存地址会被放在bss上的add全局变量里，但并没有什么用：
  2. 随后创造data heap，这里有一个漏洞就是*((_BYTE *)data_heap + v0) = 0;中的v0它来自于第一次输入的size，所以只要第一次输入的size大于后续输入的真正calloc的size，就可完成任意相对位置的写入”\x00”也就是null bytes的操作。
  3. 然后是创造一个dummy chunk，这个主要是为后面的二叉树（树节点的结构后面会提到）服务的，当作叶节点，作为遍历的终点，虽然这里对申请的chunk做了一个随机化处理，就是说chunk开头会随机填充一些长度的0xDA，末尾也会随机填充一些长度的0x42，中间的0x28字节其实才是真正的有效chunk，这里可以通过malloc_chunk看出来（不过显然这里对我们的利用没有影响）。
  4. 接下来就是对系统调用的限制，可以通过seccomp-tools进行自动分析（这里我后面忘了不可以用system调用了，浪费很多时间）：
  5. 后面就是功能菜单了，提供Allocate和Read，Free功能其实没有：
  6. add功能其实就是根据你输入的key，分配一个0x28的chunk，然后插入到二叉树中去，注意这里v4 = _strdup((__int64)&v6);可以用来泄露栈上的libc地址： 这里就可以看出node的结构：
  7. view功能就是根据输入的key去二叉树里搜索，然后把node->data打印出来，注意这里调用之后就close(1)了，后面输出都看不见了（开始没注意还搞不清楚情况，一个劲地分析原因）：

利用思路

  • 在分配的内存需求很大时（0x200000)，系统会调用mmap分配一块紧挨着libc的内存（0x1000页对齐），根据我们上面提到的任意偏移位置写null byte，这里就能在libc里任意写一个null byte。
  • 了解到Ubuntu 17.04（Glibc 2.24）中stdin->_IO_buf_end的地址低位是”\x00”的特殊性，这里就可以覆盖stdin->_IO_buf_base低字节为”\x00”，使其指向stdin->_IO_buf_end的位置，从而后续可以往&stdin->_IO_buf_end开始的位置执行任意写。（具体原理这里就不阐述了，想要了解可以关注一下平凡路上微信公众号，往期文章的IO FILE 之任意读写中有解释。）
  • 控制了stdin以及之后的内存地址后，就可以开始展开利用了。
  • 首先利用之前提到的strdup可以用来leak libc，这里需要add两次然后再view，否则leak出来的就是PIE，没什么用。还有要注意的是，由于scanf读入的数据也会读到_IO_buf_base指向的位置，所以如果以常规的方法输入一个数字加回车很容易crash掉，这里要想办法绕一下，我也是尝试了各种输入才达到目的，其实其中的原理不是很懂，待补充。
  • leak出libc的地址之后，就可以开始把_IO_buf_end改得更大，覆盖到main_arena结构体的位置（虽然中间会经过__malloc_hook，但是由于seccomp的原因onegadget是行不通的）。所以这里的利用方法是劫持_dl_open_hook。
  • _dl_open_hook是由_dl_open触发的，会经过malloc_printerr调用到，而_dl_open_hook的结构是包含了三个函数指针dlopen_mode，dlsym，dlclose: 从文章里看到，其中当_dl_open_hook不为空时，会触发_IO_vtable_check，从而触发_dl_open_hook而跳到dlopen_mode上去执行。
  • 于是可以伪造一个unsorted bin通过unsorted bin attack在_dl_open_hook写入main_arena+0x58。
  • 这里就能得到一次执行gadget的机会，因为跳到gadget执行的时候，rax=main_arena+0x58，因此可以通过处在libc_base + 0x6ebbb的gadgetmov rdi, rax; call qword ptr [rax + 0x20];来控制rdi和rip。所以就在main_arena+0x58的位置写入libc_base + 0x6ebbb，这样下一次的gadget可以部署在[rdi+0x20]的位置上（也就是main_arena+0x78）。
  • 之后再部署下一个gadget为setcontext+51，从而控制rsp，劫持栈到可控的地方，完成后续ROP：（开始我也很好奇为什么这里是不是setcontext+53而是setcontext+51，后来发现setcontext+53会构造出\x45也就是字符E，而binary中我们选择功能菜单的时候使用的是IO_getc，它其实会不断地从stdin的缓冲区读字符来刷新_IO_read_ptr，直到与_IO_read_end相等，现在如果缓冲区有了E，那就会触发[E]xit功能，从而程序结束（其实会触发SIGSYS，因为会调用5号系统调用，而被seccomp检测。）
  • 之后就是布置ROP了，libc中的ROP简直不要太多，然而因为不能getshell，那么只能通过orw了，因为view把标准输出关闭了，所以这里write到标准输入(0)，也就是write(0, buf, len);，同样可以达到输出的目的。
  • 另外注意写payload的过程中，要保留libc上原有的一些数据，否则很容易crash，而且send完payload之后，sleep一段时间（可能是等待IO_getc刷新缓冲区）后再去触发malloc_printerr，否则可能读不到flag就crash了。

小结

  • scanf的那一部分还没有搞清楚原理，貌似可以用”\xFF”（或者其他非ASCII字符）充当EOF？
  • 有太多我不知道的东西了，开始找不到mov rdi, rax; call qword ptr [rax + 0x20];这个gadget，完全不知道_dl_open_hook是个什么东西，更不会想到就写一个null byte能有这么强大。_IO_FILE可以玩到这么花（虽然是好几年前人家玩剩下的了）。
  • 我太难了，感觉还有好多东西要学啊。
